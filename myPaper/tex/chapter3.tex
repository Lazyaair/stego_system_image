\section{系统设计}\label{chap:design}

本章从需求分析出发，详细阐述系统的总体架构设计、服务端设计、客户端设计和安全设计方案。

\subsection{需求分析}

\subsubsection{功能需求}

通过对可证安全图像隐写通信系统的应用场景分析，确定系统的功能需求如下：

\begin{enumerate}
    \item \textbf{用户管理}：支持用户注册、登录、注销，每个用户拥有唯一的邀请码标识。
    \item \textbf{好友管理}：通过邀请码添加好友，支持好友请求的发送、接受和拒绝。
    \item \textbf{实时聊天}：支持一对一文本消息的即时收发，消息持久化存储于本地。
    \item \textbf{隐写消息}：在聊天过程中支持隐写模式，可将文本消息嵌入图像中发送，接收方可提取和查看隐藏的消息。
    \item \textbf{独立隐写工具}：提供独立的消息嵌入和提取页面，支持用户手动输入密钥进行隐写操作。
    \item \textbf{跨平台支持}：系统需同时支持 Android 和 Web 两个平台，不同平台的用户可以互相通信。
\end{enumerate}

\subsubsection{非功能需求}

\begin{enumerate}
    \item \textbf{安全性}：隐写算法须满足可证安全条件；用户认证采用 JWT 令牌机制；密码存储使用安全的哈希算法。
    \item \textbf{实时性}：消息收发延迟应在秒级以内（不含隐写计算时间）。
    \item \textbf{可靠性}：当用户离线时，消息应缓存于服务端，待用户上线后自动投递。
    \item \textbf{易用性}：客户端界面简洁直观，隐写功能与普通聊天无缝集成。
\end{enumerate}

\subsection{系统总体架构}

系统采用客户端--服务端（C/S）架构，由服务端、Android 客户端和 Web 客户端三部分组成。系统总体架构如\cref{fig:architecture}所示。

    \caption{系统总体架构}
    \label{fig:architecture}

各组件的职责分工如下：

\begin{itemize}
    \item \textbf{服务端}：负责用户认证与管理、好友关系维护、WebSocket 连接管理与消息路由、离线消息缓存、Pulsar 隐写算法的执行（嵌入与提取计算）。
    \item \textbf{Android 客户端}：提供移动端的用户界面，包括登录注册、联系人管理、聊天界面和隐写工具，使用 Room 数据库本地存储聊天记录。
    \item \textbf{Web 客户端}：提供浏览器端的用户界面，功能与 Android 客户端对齐，使用 IndexedDB 本地存储聊天记录。
\end{itemize}

服务端与客户端之间通过两种通信方式交互：

\begin{itemize}
    \item \textbf{RESTful API（HTTP）}：用于用户注册登录、隐写嵌入提取、邀请码查询等请求-响应式操作。
    \item \textbf{WebSocket}：用于实时消息推送、好友请求通知、在线状态变化等需要双向实时通信的场景。
\end{itemize}

\subsubsection{离线消息队列设计}

当目标用户不在线时，服务端将消息暂存于 \texttt{message\_queue} 表中，并设置过期时间（默认7天）。当用户重新连接 WebSocket 时，服务端自动查询并投递该用户的所有未过期离线消息，投递完成后从队列中删除。同时，服务端启动定时任务每5分钟清理过期消息。

\subsubsection{隐写服务集成设计}

隐写服务通过 \texttt{PulsarService} 类封装 Pulsar 算法，采用单例模式管理模型实例。每个（模型、密钥）组合对应一个独立的 Pulsar 实例，实例创建时自动完成模型加载和区域估计。嵌入和提取操作通过服务类的类方法调用，支持多种预训练模型（CelebA-HQ、Church、Bedroom、Cat）。

\subsubsection{Web 客户端架构}

Web 客户端采用 Vue.js 3 组合式 API 开发，使用 Pinia 进行集中式状态管理。

\begin{itemize}
    \item \textbf{视图层}：Vue 单文件组件（SFC），包括 LoginView、ContactsView、ChatView、EmbedView、ExtractView 等。
    \item \textbf{状态管理层}：Pinia Store 管理全局状态，包括 AuthStore（认证状态）、ChatStore（聊天与 WebSocket 连接）、ContactStore（联系人管理）。
    \item \textbf{数据持久层}：使用 IndexedDB（通过 Dexie.js 封装）存储聊天消息和联系人数据。
\end{itemize}

\subsubsection{本地存储设计}

Android 客户端的 Room 数据库包含以下实体：

\begin{itemize}
    \item \textbf{Contact}：联系人信息（用户ID、用户名、邀请码、添加时间）。
    \item \textbf{ChatMessage}：聊天消息（消息ID、发送者、接收者、内容、时间、消息类型、隐写图像数据）。
    \item \textbf{PendingRequest}：待处理的好友请求。
\end{itemize}

Web 客户端的 IndexedDB 数据库结构与之对应，使用 Dexie.js 定义 contacts 和 messages 两个对象存储。

\subsection{安全设计}

\subsubsection{JWT 认证机制}

系统使用 JSON Web Token（JWT）进行用户身份认证。用户登录成功后，服务端签发包含用户ID和用户名的 JWT 令牌，有效期为7天。客户端在后续的 HTTP 请求中通过 Authorization 头部携带令牌，WebSocket 连接通过 URL 查询参数传递令牌。服务端使用 HMAC-SHA256 算法验证令牌的完整性和有效性。

\subsubsection{密码安全}

用户密码使用 PBKDF2-HMAC-SHA256 算法进行哈希存储，迭代次数为100,000次，每个用户使用随机生成的 UUID 作为盐值。这种方案有效防止了彩虹表攻击和暴力破解。

\subsubsection{隐写密钥派生}

在隐写聊天模式下，系统自动使用通信双方的邀请码拼接作为隐写密钥。具体而言，密钥 $K = \text{sender\_invite\_code} \| \text{receiver\_invite\_code}$，其中 $\|$ 表示字符串拼接。这种设计保证了每一对通信用户拥有唯一的密钥，且双方都能独立计算出相同的密钥。

\subsubsection{单端登录策略}

为防止多设备同时登录引发的数据一致性问题，系统实施单端登录策略。当用户在新设备登录时，服务端向旧连接发送 \texttt{kicked} 消息通知其被踢出，旧客户端收到通知后清除本地令牌并跳转到登录页面。5秒后服务端强制关闭旧连接作为兜底措施。

本章从功能需求和非功能需求出发，设计了系统的总体架构，详细阐述了服务端的 API 接口、数据库、WebSocket 通信协议、离线消息队列和隐写服务的设计方案，以及客户端的架构和本地存储设计。同时，从 JWT 认证、密码安全、隐写密钥派生和单端登录四个方面介绍了系统的安全设计。
